# Fase 7b — E2E (Playwright) — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Suíte Playwright cobrindo os fluxos críticos + casos de borda (RBAC/estado/convite) sobre o app existente, rodável localmente e pronta para um job de CI efêmero.

**Architecture:** Pacote `e2e/` dedicado. O `playwright.config` sobe api(3000)+web(3001) via `webServer`; Postgres+MinIO vêm de um `docker-compose` (dev e E2E local). Isolamento por **usuário novo por teste** (e-mail aleatório, sem reset de DB). Pré-requisitos semeados via **API real** com o cookie do usuário (`page.request`). Seletores acessíveis (`getByRole/Label/Text`) + poucos `data-testid` em itens de lista.

**Tech Stack:** `@playwright/test`, Bun workspaces, Vite (web 3001, proxy `/api`→3000), Elysia (api 3000), Better Auth (email/senha), Postgres + MinIO.

**Spec:** `docs/superpowers/specs/2026-06-14-fase-7b-e2e-design.md`

**Git:** branch `feat/fase-7b-e2e` a partir de `develop`; commit por tarefa; no fim, tudo verde → merge em `develop` e marcar a 7b no ROADMAP.

**Importante (fronteira CD):** este plano **não altera `.github/workflows`**. Entrega o harness + specs + um guia do que o job de CI precisa; o job é escrito pelo usuário (trilha CD).

**Característica do E2E:** caracteriza comportamento **existente** — o "teste falha primeiro" não se aplica (a feature já existe). Fluxo: escrever o spec → rodar → passar. Se falhar, é bug real ou seletor: investigar com `superpowers:systematic-debugging`.

```ts
import type { APIRequestContext, Browser, Page } from "@playwright/test";
import { expect } from "@playwright/test";

const here = path.dirname(fileURLToPath(import.meta.url));

/** Registra um usuário novo pela UI e espera cair no dashboard. Retorna o e-mail. */
export async function signup(page: Page, email = randomEmail()): Promise<string> {
  await page.goto("/login");
  await page.getByRole("button", { name: /Não tem conta\? Cadastre-se/i }).click();
  await page.locator("#name").fill("Usuário E2E");
  await page.locator("#email").fill(email);
  await page.locator("#password").fill("password123");
  await page.getByRole("button", { name: /^Criar conta$/ }).click();
  await page.waitForURL("**/"); // dashboard
  return email;
}

/** Cria um usuário isolado em seu próprio contexto (cookies próprios). */
export async function newUser(
  browser: Browser
): Promise<{ page: Page; email: string; close: () => Promise<void> }> {
  const context = await browser.newContext();
  const page = await context.newPage();
  const email = await signup(page);
  return { page, email, close: () => context.close() };
}

interface SeedSchedule {
  mode: "auto" | "custom";
  totalAmountCents?: number;
  installmentsCount?: number;
  firstDueDate?: string;
  installments?: { amountCents: number; dueDate: string }[];
}

/** Cria um contrato via API real (cookie da sessão). Retorna { id }. */
export async function seedContract(
  request: APIRequestContext,
  opts: {
    title?: string;
    ownerRole?: "buyer" | "seller";
    requiresConfirmation?: boolean;
    schedule?: SeedSchedule;
  } = {}
): Promise<{ id: string }> {
  const res = await request.post("/api/contracts", {
    data: {
      title: opts.title ?? "Contrato E2E",
      ownerRole: opts.ownerRole ?? "buyer",
      requiresConfirmation: opts.requiresConfirmation ?? false,
      schedule:
        opts.schedule ??
        ({
          mode: "auto",
          totalAmountCents: 300_000,
          installmentsCount: 3,
          firstDueDate: "2026-09-10",
        } satisfies SeedSchedule),
    },
  });
  expect(res.ok()).toBeTruthy();
  return (await res.json()) as { id: string };
}

/** Lê o detalhe do contrato (parcelas, participantes) via API. */
export async function getContract(request: APIRequestContext, id: string) {
  const res = await request.get(`/api/contracts/${id}`);
  expect(res.ok()).toBeTruthy();
  return res.json();
}

/** Adiciona participante e gera convite travado por e-mail. Retorna { token, participantId }. */
export async function seedInvite(
  request: APIRequestContext,
  contractId: string,
  opts: { displayName: string; role: "buyer" | "seller" | "viewer"; email: string }
): Promise<{ token: string; participantId: string }> {
  const add = await request.post(`/api/contracts/${contractId}/participants`, {
    data: { displayName: opts.displayName, role: opts.role },
  });
  expect(add.ok()).toBeTruthy();
  const participant = (await add.json()) as { id: string };
  const inv = await request.post(
    `/api/contracts/${contractId}/participants/${participant.id}/invite`,
    { data: { email: opts.email } }
  );
  expect(inv.ok()).toBeTruthy();
  const body = (await inv.json()) as { token: string };
  return { token: body.token, participantId: participant.id };
}
```

## Tasks 7–12: specs por área (seguir o padrão das Tasks 5–6)

Cada task: criar o arquivo, implementar os cenários abaixo (imports `expect/test` do
`@playwright/test` + helpers de `../fixtures`), rodar `bun run e2e specs/<arquivo>` até passar
(ajustando seletores contra os componentes reais), e commitar. Use `newUser(browser)` para
cenários de 2 usuários e `seedContract`/`seedInvite`/`getContract` para pré-requisitos via API.

### Task 7 — `payments.spec.ts`
Contrato **com confirmação** entre 2 usuários (comprador = dono; vendedor = convidado aceito —
use `seedInvite` + aceitar pela UI, ou semeie ambos os lados via API e logue cada um).
- **Enviar comprovante:** comprador abre a parcela (`?installment=<id>`), faz `setInputFiles` no
  input `getByLabel("Comprovante")` com `PROOF_PDF`, clica "Enviar comprovante" → status muda para
  "aguardando".
- **Confirmar:** vendedor abre a parcela → "Confirmar pagamento" (com diálogo) → status
  "confirmada"/"paga"; link de recibo aparece.
- **Contestar + reenviar:** vendedor "Contestar" (preenche `#dispute-reason`) → "contestada";
  comprador reenvia comprovante.
- **Sem confirmação:** contrato `requiresConfirmation:false` → comprador "Marcar como paga" → "paga".
- **Recibo ausente quando não-paga:** numa parcela pendente, o link "Baixar recibo" não aparece.

### Task 8 — `rbac.spec.ts`
Dois usuários (A dono, B estranho).
- A cria contrato (`seedContract`). B (outro contexto) faz `page.goto('/contracts/<idDeA>')` →
  vê estado de "não encontrado"/erro (não vaza); **não** vê o título do contrato de A.
- B tenta o deep-link `'/contracts/<idDeA>?installment=<algumId>'` → idem bloqueado.

### Task 9 — `invites.spec.ts`
Dois usuários.
- **Aceitar (e-mail certo):** A cria contrato + `seedInvite(role:'seller', email: emailDeB)`. B
  (e-mail = o convidado) abre `/invites/<token>` → "Aceitar convite" → vai pro contrato e o vê na
  lista.
- **E-mail errado:** registra B com e-mail diferente do convite → abre o token → vê "Este convite
  é para outro e-mail"; botão "Aceitar convite" não disponível.
- **Já participante:** B aceita; reabre o token → "Você já participa deste contrato".
- **Convite já usado:** após B aceitar, um terceiro com o mesmo e-mail não consegue (mensagem de
  convite indisponível/usado).
- **Gerenciar:** A abre "Gerenciar", remove um participante (diálogo de confirmação) e troca o
  papel de outro via `#role-<participantId>`.

### Task 10 — `notifications.spec.ts`
Dois usuários, contrato **com confirmação** (A comprador, B vendedor vinculado).
- A envia comprovante (via UI ou seed do proof + ação). B abre o app → o **sininho** mostra
  contador > 0; B vai em `/notifications` → vê "Novo comprovante para confirmar"; clicar no item
  (`data-testid="notification-<id>"`) **navega** pro `/contracts/<id>?installment=<id>` e abre o
  drawer.
- "Marcar todas como lidas" → contador zera.

### Task 11 — `lgpd.spec.ts`
- **Exportar:** usuário com 1 contrato → `/settings` → clicar "Exportar meus dados" dispara
  download (`page.waitForEvent('download')`); o arquivo é `quitto-meus-dados.json`.
- **Excluir conta:** `/settings` → "Excluir conta" → digitar texto errado no `#confirm-phrase`
  mantém "Excluir definitivamente" **desabilitado**; digitar `EXCLUIR` habilita → confirmar →
  cai no `/login`.

### Task 12 — `dashboard.spec.ts`
- **Empty state:** usuário novo em `/` → vê `data-testid="dashboard-empty-state"` (adicione o

## Notas para o executor

- **Caracterização, não TDD:** os fluxos já existem; specs devem passar de primeira. Falha = bug
  real (use `systematic-debugging`) ou seletor errado (ajuste contra o componente real).
- **Seletores acessíveis primeiro** (`getByRole/Label/Text`); `data-testid` só nos itens de lista
  da Task 4 + os pontuais citados (`stat-to-pay`, `dashboard-empty-state`). Ao adicionar um testid
  novo, faça-o no componente do `apps/web` no mesmo commit do spec que o usa.
- **2 usuários:** sempre via `newUser(browser)` (contextos separados = cookies isolados). Nunca
  compartilhe página entre os dois papéis.
- **Seed via API real:** `page.request.post('/api/...')` herda o cookie da sessão da página. Use
  caminhos relativos (passam pelo proxy do Vite). Confirme os shapes de retorno nos módulos da API.
- **Upload:** `page.getByLabel('Comprovante').setInputFiles(PROOF_PDF)`.
- **Estabilidade:** prefira `await expect(...).toBeVisible()` / `waitForURL` a `waitForTimeout`.
  `fullyParallel` ligado — cada teste com seu usuário, sem estado compartilhado.
- **Não toque em `.github/workflows`** — o job é do usuário (trilha CD).
