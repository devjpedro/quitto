# Guia de estudo — job de E2E no CI (Fase 7b)

> **Escopo / fronteira CD.** Este documento é **material de estudo**, não o arquivo de
> workflow final. Pela mesma regra da fase CD (GitHub Actions é dono do deploy e do CI —
> **o usuário escreve o YAML, o Claude ensina**), o que segue descreve **o que** o job `e2e`
> precisa, com trechos **ilustrativos**. Adapte e escreva o `.github/workflows/ci.yml` você mesmo.

## O que a suíte é

A Fase 7b entregou a suíte Playwright em `e2e/` (workspace `@quitto/e2e`), com **8 spec files
= 25 testes** cobrindo os fluxos críticos: `smoke`, `auth`, `contracts`, `payments` (pagar /
confirmar / contestar), `rbac` (no-leak), `invites` (incl. e-mail errado / convite já usado),
`notifications` (+ deep-link), `lgpd` e `dashboard`.

O pacote sobe **api + web** sozinho via o `webServer` do Playwright
(`e2e/playwright.config.ts`) — não há passo manual de "subir os serviços" no job. O `webServer`
roda `bun run --filter @quitto/api dev` e `bun run --filter @quitto/web dev` e espera os
health-checks (`/api/ping` e `http://localhost:3001`).

### Isolamento de estado (sem endpoints de teste, sem reset de DB)

Cada teste **cria um usuário novo** (sign-up real com e-mail aleatório) e **semeia os
pré-requisitos pela API real**, usando o cookie de sessão. **Não existem endpoints
test-only**, e **não há reset de banco entre os testes** — o isolamento vem do usuário novo +
RBAC por contrato (um usuário não enxerga dados do outro). Consequência prática para o CI: o job
só precisa de **um** Postgres limpo no começo (migrado), nada de truncate entre specs.

## Anatomia do job `e2e`

### `services` — iguais ao job `verify` de `ci.yml`

- **Postgres 16** (`postgres:16`) com `POSTGRES_PASSWORD`, `POSTGRES_DB: quitto`, porta
  `5432`, health-check `pg_isready`.
- **MinIO** com bucket auto-criado `quitto-proofs` (env `MINIO_DEFAULT_BUCKETS`), porta `9000`.
  **Atenção à imagem** — ver gotcha 2 abaixo.

### `env` — bloco idêntico ao `verify`

```yaml
# ilustrativo — copie do bloco env do job verify em ci.yml
DATABASE_URL: postgres://postgres:postgres@localhost:5432/quitto
BETTER_AUTH_SECRET: ci-secret-ci-secret-ci-secret-ci-secret   # >= 32 chars
BETTER_AUTH_URL: http://localhost:3000
WEB_ORIGIN: http://localhost:3001
S3_ENDPOINT: http://localhost:9000
S3_REGION: us-east-1
S3_BUCKET: quitto-proofs
S3_ACCESS_KEY_ID: minio
S3_SECRET_ACCESS_KEY: minio12345
```

### Steps, em ordem

1. `actions/checkout@v4`
2. `oven-sh/setup-bun@v2` (`bun-version: latest`)
3. `bun install --frozen-lockfile`
4. **Rodar as migrations** — `bun run --filter @quitto/api db:migrate` (forma que funciona —
   ver gotcha 1).
5. **Instalar o browser** — `cd e2e && bunx playwright install --with-deps chromium`
   (`--with-deps` traz as libs de sistema do Chromium no runner).
6. **Rodar a suíte** — `cd e2e && bun run e2e`. Com `CI=true` (setado automaticamente pelo
   GitHub Actions), o `playwright.config.ts` liga `retries: 1`, troca o reporter para `github`
   e **desliga** `reuseExistingServer` — ou seja, o `webServer` **sempre** sobe api+web do zero
   nesse runner.

### Artefato em caso de falha

Suba o `e2e/playwright-report` só quando o job falhar, pra inspecionar o trace:

```yaml
# ilustrativo
- name: upload playwright report
  if: failure()
  uses: actions/upload-artifact@v4
  with:
    name: playwright-report
    path: e2e/playwright-report
    retention-days: 7
```

### Job SEPARADO do `verify`

Mantenha `e2e` como um **job distinto** do `verify` (em paralelo ou em sequência), **não** como
mais alguns steps dentro do `verify`. O `verify` é o portão rápido (lint/typecheck/test/build);
o E2E é mais pesado (sobe api+web, baixa o Chromium). Misturar os dois deixa o gate rápido
lento. Se quiser que o E2E só rode após o gate passar, use `needs: verify`.

## Dois gotchas descobertos na Fase 7b (IMPORTANTE)

Chame estes dois pontos explicitamente — eles afetam tanto o **novo job `e2e`** quanto o
**job `verify` que já existe** em `ci.yml`. Sem isso, o CI pode quebrar silenciosamente.

### 1. Sintaxe do `--filter` no bun

`bun --filter <pkg> run <script>` (o filtro **antes** do `run` posicional) **FALHA** com
`No packages matched the filter` em versões recentes do bun (observado no **bun 1.3.x**).

- ✅ Forma que funciona: **`bun run --filter @quitto/api db:migrate`** (`run` primeiro, depois
  `--filter`).
- O `e2e/playwright.config.ts` **já usa a forma boa** (`bun run --filter @quitto/web dev`) no
  `webServer` — por isso a suíte sobe.
- ⚠️ **O job `verify` em `ci.yml` ainda usa a forma antiga**:
  `bun --filter @quitto/api run db:migrate` (linha do passo de migração). Em bun novo isso vai
  dar `No packages matched the filter`. **Atualize para a forma boa**
  (`bun run --filter @quitto/api db:migrate`) ao mexer no workflow.

### 2. Imagem do MinIO (`bitnami/minio:latest` foi removida)

A Docker Hub **removeu** as tags `latest` gratuitas do namespace `bitnami/*` (movidas para
`bitnamilegacy/*`). O `docker-compose.yml` local **já** aponta para
**`bitnamilegacy/minio:latest`** (drop-in, mesmas envs / auto-bucket).

- O `services.minio` no CI deve fazer o mesmo: use **`bitnamilegacy/minio:latest`** (ou pin numa
  versão real publicada de `bitnami/minio`). Caso contrário o **pull da imagem falha**.
- ⚠️ Isso vale para **ambos** os jobs: o `verify` que já existe (`ci.yml` ainda usa
  `bitnami/minio:latest`) **e** qualquer novo job `e2e`.

## Esqueleto ilustrativo do job (NÃO é o arquivo final)

```yaml
# ILUSTRATIVO — escreva o seu em .github/workflows/ci.yml
  e2e:
    runs-on: ubuntu-latest
    # needs: verify   # opcional, se quiser rodar após o gate rápido
    services:
      postgres:
        image: postgres:16
        env: { POSTGRES_PASSWORD: postgres, POSTGRES_DB: quitto }
        ports: ["5432:5432"]
        options: >-
          --health-cmd pg_isready --health-interval 10s
          --health-timeout 5s --health-retries 5
      minio:
        image: bitnamilegacy/minio:latest   # gotcha 2
        env:
          MINIO_ROOT_USER: minio
          MINIO_ROOT_PASSWORD: minio12345
          MINIO_DEFAULT_BUCKETS: quitto-proofs
        ports: ["9000:9000"]
    env:
      # ... mesmo bloco env do verify (ver acima) ...
    steps:
      - uses: actions/checkout@v4
      - uses: oven-sh/setup-bun@v2
        with: { bun-version: latest }
      - run: bun install --frozen-lockfile
      - run: bun run --filter @quitto/api db:migrate    # gotcha 1
      - run: cd e2e && bunx playwright install --with-deps chromium
      - run: cd e2e && bun run e2e
      - if: failure()
        uses: actions/upload-artifact@v4
        with: { name: playwright-report, path: e2e/playwright-report }
```
