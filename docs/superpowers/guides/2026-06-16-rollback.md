# Guia — Rollback de produção (emergência)

Quando um deploy sobe quebrado, **voltar pra versão anterior é rápido** — Fly e Vercel guardam os
releases. Este é o runbook pra usar no susto. Regra de ouro: **reverter primeiro, investigar depois.**

## API (Fly) — `usequitto-api`

```fish
# listar releases (o topo é o atual)
fly releases -a usequitto-api

# reverter pro release imediatamente anterior
fly releases rollback -a usequitto-api

# ou voltar pra uma imagem específica de um release antigo
fly deploy -a usequitto-api --image <registry.fly.io/usequitto-api:deployment-...>
```

Confirme: `curl https://usequitto-api.fly.dev/api/ping` → `{"status":"ok"}`.

## Web (Vercel) — `usequitto`

**Dashboard (mais rápido):** projeto `usequitto` → **Deployments** → escolha um deploy de produção
**anterior que estava bom** → menu `⋯` → **Promote to Production**. É instantâneo (só re-aponta o
alias, sem rebuild).

**CLI:**
```fish
cd apps/web
vercel ls --scope devjpedro1s-projects          # lista deployments (pega a URL de um bom)
vercel promote <url-do-deploy-bom> --scope devjpedro1s-projects
```

Confirme: `https://usequitto.vercel.app` carrega e `/api/me` responde (200/401, não erro).

## Banco (Neon) — o cuidado importante

Migrations são **só pra frente**. Não existe "rollback" trivial de uma migration que **dropou
coluna/tabela** — os dados já foram. Por isso:

- **Migrations aditivas primeiro** (adicionar coluna nullable, backfill, depois tornar obrigatória)
  — assim a versão antiga do código continua funcionando se você reverter o deploy.
- Se uma migration ruim subiu, o conserto normalmente é **uma nova migration corretiva**, não desfazer.
- O Neon tem **branching/restore por ponto no tempo** (point-in-time) no painel — use só em último
  caso de perda de dado, ciente de que reverte o banco inteiro.

## Ordem numa reversão típica

1. Reverta a **API** (Fly rollback) e o **web** (Vercel promote) pra última versão boa.
2. Confirme com o smoke (`/api/ping` + site).
3. **Só então** investigue o que quebrou, corrija na `develop`, e suba de novo pelo pipeline.

## Pra "testar uma vez" (Marco 6 do guia de CD)

Reverta a API de propósito (`fly releases rollback`) num momento tranquilo, veja o `/api/ping`
seguir 200, e depois redeploy pela `main`. Assim você já fez o movimento antes de precisar dele.
